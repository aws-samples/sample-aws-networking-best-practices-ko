# 새로운 모범 사례 제안 {#propose-a-new-best-practice}

AWS 네트워킹 가이드를 개선할 아이디어가 있으신가요? 여러분의 의견을 환영합니다! 이 페이지에서는 전체 커뮤니티에 도움이 되는 새로운 모범 사례를 제안하는 방법을 안내합니다.

## 빠른 개요 {#quick-overview}

제안서를 제출하기 전에:

1. **실제로 새로운 모범 사례인지 확인하세요** ([버그 보고]가 아닌지 확인)

2. **기존 솔루션을 조사하고** 참조하세요

3. **AWS re:Post에서 커뮤니티와 논의하세요**

4. **템플릿을 사용하여 상세한 제안서를 제출하세요**

시작할 준비가 되셨나요? 전체 프로세스는 계속 읽어보세요.

  [버그 보고]: report-a-correction.md

## 좋은 제안서의 조건 {#what-makes-a-good-proposal}

### 정말 새로운 모범 사례인가요? {#is-this-really-a-new-best-practice}

새로운 모범 사례는 커뮤니티에 상당한 가치를 더해야 합니다. 스스로에게 물어보세요:

* 일반적인 네트워킹 문제를 해결하나요?

* 특정 조직뿐만 아니라 여러 조직에 도움이 되나요?

* 네트워킹에 특화된 가이드인가요? (일반적인 AWS 조언이 아닌)

**확실하지 않으신가요?** 오류나 불일치를 보고하는 경우라면 (새 모범 사례 대신) [수정 사항 보고](report-a-correction.md)를 사용하세요.

### 충분히 조사하세요 {#do-your-research}

* AWS 문서, 블로그, 포럼에서 기존 솔루션을 검색하세요

* 기존 가이드를 참조하고 제안이 어떻게 개선되는지 설명하세요

* 관련 주제에 대한 [커뮤니티 토론](https://repost.aws/topics/TA-2izgznkTKe0-VdIELPAgg/networking-content-delivery)을 확인하세요

## 이슈 템플릿 {#issue-template}

제안서를 제출할 준비가 되셨나요? [이 템플릿](https://github.com/aws/aws-networking-best-practices/issues/new?template=new-best-practice.md)을 사용하여 필요한 모든 정보를 포함하세요:

* 제목
* 배경 <small>선택 사항</small>
* 설명
* 관련 링크
* 사용 사례
* 시각 자료 <small>선택 사항</small>

### 제목 {#title}

좋은 제목은 짧고 명확해야 합니다. 커뮤니티가 제목만으로도 잠재적인 영향과 이점을 파악할 수 있도록 아이디어를 한 문장으로 요약해야 합니다.

| <!-- --> | 예시  |
| -------- | -------- |
| :material-check:{ style="color: #4DB6AC" } **명확함** | SD-WAN integration with TGW |
| :material-close:{ style="color: #EF5350" } **장황함** | Add a feature where you describe how to integrate various SD-WAN options with a Transit Gateway |
| :material-close:{ style="color: #EF5350" } **불명확함** | Improve TGW |
| :material-close:{ style="color: #EF5350" } **무의미함** | Help |

### 배경 <small>선택 사항</small> { #context }

아이디어를 설명하기 전에, 달성하려는 목표를 이해할 수 있도록 추가적인 배경 정보를 제공할 수 있습니다. 제안된 모범 사례를 사용하는 상황과 관련이 있다고 _생각하는_ 내용을 설명하세요. 실제 모범 사례 내용은 여기에 작성하지 마세요.

> **이 항목이 도움이 되는 이유**: 일부 아이디어는 특정 고객, 환경, 또는 엣지 케이스(예: 수천 개의 VPC가 포함된 네트워크)에서만 유용할 수 있습니다. 배경 정보가 있으면 모범 사례 가이드의 우선순위를 보다 정확하게 결정할 수 있습니다.

### 설명 {#description}

다음으로, 아이디어에 대한 상세하고 명확한 설명을 제공하세요. 아이디어가 AWS 네트워킹 아키텍처 가이드와 어떤 관련이 있으며, 다른 AWS 문서나 블로그 게시물이 아닌 이곳에 추가되어야 하는 이유를 설명하세요.

*   **'<u>무엇</u>'을 설명하고 '<u>왜</u>'는 다루지 마세요** – 아이디어의

    이점은 다음 항목에서 다룹니다.
    제안된 솔루션을 최대한 정확하게 설명하는 데 집중하세요.

*   **간결하게 작성하세요** – 아이디어를 설명할 때 핵심만 간략히

    작성하세요. 과도하게 설명할 필요가 없습니다. 메인테이너와 향후 사용자들이 더 적게 읽어도 되는 것에 감사할 것입니다.

*   **한 번에 하나의 아이디어만** – 서로 관련 없는 여러 아이디어가 있다면

    각 아이디어에 대해 별도의 새로운 모범 사례 제안서를 작성해 주세요.

---

> **이 항목이 필요한 이유**: 제안된 변경 사항을 이해하고 평가하기 위해서는 아이디어에 대한 명확한 이해가 필요합니다. 상세하고 정확한 설명을 제공하면 댓글에서 추가 설명을 논의하는 데 소요되는 시간을 여러분과 저희 모두 아낄 수 있습니다.

### 관련 링크 {#related-links}

새로운 모범 사례 제안과 관련된 이슈, 토론, 또는 문서 섹션의 링크를 제공해 주세요. 커뮤니티 토론 게시판에서 이미 이 아이디어에 대해 논의한 경우, 해당 토론 링크도 포함해 주세요.

> **이 항목이 필요한 이유**: 관련 링크는 추가적인 맥락을 제공하여 새로운 모범 사례 제안을 포괄적으로 이해하는 데 도움이 됩니다. 또한 이전 이슈 및 토론에 대한 링크를 통해 커뮤니티에서 이미 제공된 피드백과 의견을 신속하게 평가할 수 있습니다.

### 사용 사례 {#use-cases}

새로운 모범 사례 제안이 사용자 관점에서 어떻게 작동하는지 설명하세요. 예상되는 영향은 무엇이며, 본인뿐만 아니라 다른 사용자들에게도 어떤 이점이 있나요? 얼마나 많은 사용자에게 해당되나요? 또한 다른 모범 사례와 충돌할 가능성이 있나요?

> **이 항목이 필요한 이유**: 아이디어의 사용 사례와 이점을 이해하는 것은 프로젝트와 사용자에 대한 잠재적 영향과 유용성을 평가하는 데 매우 중요합니다. 이 정보는 아이디어의 예상 가치와 프로젝트 목표와의 부합 여부를 파악하는 데 도움이 됩니다.

### 시각 자료 <small>선택 사항</small> { #visuals }

이제 잠재적인 사용 사례와 관련 링크를 포함한 명확하고 상세한 아이디어 설명이 완성되었습니다. 스케치, 스크린샷, 목업, 외부 자료 등의 시각 자료가 있다면 이 섹션에 첨부해 주세요.

**파일을 여기에 드래그 앤 드롭하거나 외부 자료 링크를 포함할 수 있습니다.**

또한 다른 공개 웹사이트에서 이 변경 사항, 기능, 또는 개선 사항이 사용된 것을 본 적이 있다면, 해당 예시를 제시하고 어떻게 구현 및 적용되었는지 설명해 주세요.

> **이 항목이 도움이 되는 이유**: 그림과 시각 자료는 메인테이너가 아이디어를 더 잘 이해하고 구체화하는 데 도움이 됩니다. 스케치나 다이어그램은 텍스트만으로는 전달하기 어려운 세부 사항과 명확성을 추가로 제공할 수 있습니다. 또한 다른 공개 웹사이트에서 아이디어가 어떻게 문서화되었는지 확인하면 잠재적인 영향과 실현 가능성을 파악하는 데 도움이 되어, 메인테이너가 새로운 모범 사례 제안을 평가하고 분류하는 데 도움이 됩니다.

**이후 과정은 저희가 진행하겠습니다.**

## 제출 후 절차 {#after-you-submit}

제안서를 제출한 후:

* **인내심을 가지세요** - 커뮤니티 영향을 검토하는 데 시간이 걸립니다

* **토론에 참여하세요** - 질문과 피드백에 응답하세요

* **변경에 열린 자세를 가지세요** - 커뮤니티 의견을 통해 아이디어가 발전할 수 있습니다

* **계속 참여하세요** - 승인된 제안의 구현을 돕는 것을 고려해 보세요

## 제안이 거절된 경우 {#if-your-proposal-is-rejected}

거절이 실망스러울 수 있다는 것을 이해합니다. 저희는 다음 기준을 바탕으로 제안을 평가합니다:

* 커뮤니티 이점 및 광범위한 적용 가능성

* 프로젝트 목표와의 부합 여부

* 구현 및 유지 관리 노력

* 명확성 및 완성도
